Token Claims — Three-Stage Evolution

Watch how identity transforms across the exchange: user identity is preserved, AI agent identity is added via the act claim

① Org ID Token iss: Org AS
"iss": "zelemon.oktapreview.com"
"sub": "00u5h2mvt28EZy1DA0x7"
"aud": "0oadzavko8niM4W3l0x7"
"cid": "0oadzavko8niM4W3l0x7"
"name": "Vincent S."
— no act claim yet —
— no expenses scopes —
Key: User identity only. Issued to Web App.
② ID-JAG iss: Org AS · 5 min TTL
"iss": "zelemon.oktapreview.com"
"sub": "00u5h2mvt28EZy1DA0x7"
"act": {"sub": "wlpe0rmyb2XBLrWCD0x7"}
"aud": "ausdo82jknZLNiOmA0x7"
"scp": "expenses:read write"
expires_in: 300s
Key: act.sub = AI Agent added. Bridge token.
③ Expenses Access Token iss: Custom AS
"iss": "…/oauth2/ausdo82…"
"sub": "00u5h2mvt28EZy1DA0x7"
"act": {"sub": "wlpe0rmyb2XBLrWCD0x7"}
"client_name": "Expenses AI Agent"
"scp": ["expenses:read",
"expenses:write",
"expenses:delete"]
Key: Full delegation. Used to call Expenses API.
💡 Key insight: sub always identifies the user (vst@okta.com). act.sub identifies the AI Agent. Any action in Okta system logs is attributed to both: "AI Agent wlpe0rmyb2X acting on behalf of user 00u5h2mvt28". This is the core of Okta Cross-App Access governance.

Delegated identity, not shared keys

Okta as the cryptographic trust broker for every agent hop

⚠️ Why it matters

Static credentials turn one compromised agent into an admin across everything it touches — internal APIs, SaaS apps, MCP servers, and other agents alike.

🏛 How Okta solves it

Okta brokers every agent connection. We validate the user, validate the agent, and issue a scoped short-lived token carrying both as cryptographic proof — so every hop can verify who's acting on whose behalf.

✅ What you get
  • →Cross-app access across authorization servers, no shared keys
  • →Every token carries user + agent as cryptographic delegation
  • →Revoke one agent without disrupting the rest
